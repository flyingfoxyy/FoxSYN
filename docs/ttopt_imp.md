# `ttopt` 扩展实现方案整理

本文整理三种用于突破 `ttopt` 当前小 support 限制、并尝试优化更大逻辑函数的实现方案：

1. `windowed ttopt`
2. `simulation + SAT guided ttopt`
3. `cut-based partial-support ttopt`

目标不是简单把 truth table 支持变量上限从 `16` 硬改大，而是设计更可扩展的优化路径，在较大逻辑锥上获得更好的 QoR。


## 1. 背景

当前 `ttopt` 的基本思路是：

1. 为一组输出收集 support。
2. 计算 truth table。
3. 通过 BDD 变量重排减小表示规模。
4. 再由该表示重建 AIG。

其核心实现位于：

- [src/abc/src/aig/gia/giaTtopt.cpp](/home/longfei/FoxSYN/src/abc/src/aig/gia/giaTtopt.cpp#L1109)

这种方法的瓶颈在于：

- truth table 的规模随变量数按 `2^n` 增长
- BDD 构建、交换、重排都依赖这个表示
- support 太大时，时间和内存都会快速失控

因此，优化大函数时，核心问题不是“如何允许更大完整 support”，而是：

- 如何在不显式枚举完整大 truth table 的前提下，仍然获得高质量优化


## 2. 方案一：`windowed ttopt`

### 2.1 基本思路

不要对节点或输出组的完整 support 直接做 truth table，而是先截取一个局部 logic window，然后只在这个 window 内应用 `ttopt` 思路。

也就是说：

1. 选取一个局部窗口
2. 为窗口建立边界输入和边界输出
3. 仅对窗口内部功能构造 truth table / BDD
4. 进行变量重排和 AIG 重建
5. 用优化后的窗口替换原子网

### 2.2 为什么它能突破 support 限制

因为完整 support 可能很大，但一个精心选择的 window 往往只暴露出有限个边界输入。  
只要窗口边界输入规模控制在一个可接受范围内，就仍然可以使用现有 `ttopt` 的核心流程。

换句话说，它不是让算法“直接处理大 support”，而是：

- 把大函数切成多个可处理的小函数窗口

### 2.3 优点

- 可以较多复用现有 `ttopt` 代码
- 比纯 cut 方法看到更完整的局部逻辑结构
- 对多输出局部区域更友好
- 工程风险较低，比较适合在现有代码上演进

### 2.4 缺点

- QoR 很依赖 window 选择质量
- 窗口太小会失去优化空间
- 窗口太大又会回到 truth table 爆炸问题
- 仍然是局部优化，不是全局函数级优化

### 2.5 QoR 预期

在三种方案中，这个方案的 QoR 预期通常居中偏上。

它通常比 `cut-based partial-support ttopt` 更好，因为：

- window 比固定 cut 更能保留真实逻辑关系
- 能利用更多局部共享和多输出关联

它通常不如 `simulation + SAT guided ttopt` 的理论上限高，因为：

- 仍然受限于局部窗口边界
- 很难利用更大上下文中的功能可替换空间

### 2.6 综合评价

如果目标是：

- 在现有 `ttopt` 基础上尽快做出可用增强版
- 希望 QoR 有明显提升
- 又不想把实现复杂度推得太高

那么 `windowed ttopt` 是最现实、最均衡的选择。


## 3. 方案二：`simulation + SAT guided ttopt`

### 3.1 基本思路

这个方案不再把完整 truth table 作为唯一中心，而是采用：

- 仿真指导候选搜索
- SAT 做功能正确性验证
- 必要时再结合局部函数表示做重建

一种典型流程是：

1. 用随机仿真或定向仿真收集节点行为签名
2. 找到可能可替换、可共享、可重构的候选区域
3. 构造候选替换网络
4. 使用 SAT 验证替换是否严格等价，或在给定 care 条件下等价
5. 接受 QoR 更优的替换

### 3.2 为什么它能突破 support 限制

因为它不要求对整个 support 显式枚举 `2^n` 模式。

它主要依靠：

- 仿真进行低成本筛选
- SAT 在必要时做精确判定

所以即便 support 很大，也仍然可以处理。

### 3.3 优点

- QoR 上限通常最高
- 可以利用比小 truth table 更大的上下文
- 更容易发现结构上不明显但功能上成立的替换
- 更容易利用 care-set、ODC、仿真模式等额外信息

### 3.4 缺点

- 实现复杂度最高
- 运行时间风险最大
- 调参与工程整合难度较大
- 如果候选生成不够好，效果可能不稳定

### 3.5 QoR 预期

在三种方案中，这个方案的 QoR 上限通常最高。

原因是：

- 它不被一个很小的固定窗口或 cut 严格限制
- 能探索更广泛的局部重构空间
- 可以在大 support 环境下继续发现有效优化机会

但它的“上限高”并不等于“最容易做出稳定收益”。

### 3.6 综合评价

如果目标是：

- 做研究型原型
- 追求更高 QoR 上限
- 能接受更长开发周期和更复杂实现

那么这是最有潜力的方案。


## 4. 方案三：`cut-based partial-support ttopt`

### 4.1 基本思路

这个方案使用有限大小的 cut 来近似观察大函数，而不是处理完整 support。

流程通常是：

1. 为节点枚举固定大小的 cut
2. 只对 cut 对应的局部函数计算 truth table
3. 在该局部函数上做变量换序或局部重建
4. 评估替换收益并决定是否回写

### 4.2 为什么它能突破 support 限制

因为 cut 的大小是人为限制的，例如：

- 6-cut
- 8-cut
- 10-cut

即使节点的完整 support 很大，cut 仍然只暴露一个较小的局部输入集，所以 truth table 成本可控。

### 4.3 优点

- 最成熟、最稳妥
- 实现复杂度相对较低
- 容易控制时间和内存
- 很适合大规模批量扫描优化

### 4.4 缺点

- 看到的函数视图最局部
- 容易错过跨 cut 的高价值优化机会
- 多输出关联利用较弱
- QoR 上限通常不如前两种方案

### 4.5 QoR 预期

在三种方案中，这个方案的 QoR 预期通常最低。

它很适合做稳定的局部改进，但不太容易产生大范围、高质量的重构收益。

### 4.6 综合评价

如果目标是：

- 快速验证思路
- 先做一个低风险版本
- 优先保证性能和实现可控性

那么它是最容易落地的第一步。


## 5. 三种方案的 QoR 预期排序

如果只看 QoR 上限，不优先考虑实现复杂度和运行时间，通常可以这样排序：

1. `simulation + SAT guided ttopt`
2. `windowed ttopt`
3. `cut-based partial-support ttopt`

原因如下：

- 第一种方案看到的上下文最大，搜索空间也最大
- 第二种方案能保留较完整的局部结构，优于固定 cut
- 第三种方案最局部，通常只能提供较保守的改进


## 6. 三种方案的综合可落地性排序

如果综合考虑：

- QoR
- 实现复杂度
- 工程风险
- 在现有 `ttopt` 基础上的可改造性

更合理的排序通常是：

1. `windowed ttopt`
2. `simulation + SAT guided ttopt`
3. `cut-based partial-support ttopt`

这里把 `windowed ttopt` 放在第一，是因为：

- 它的 QoR 预期不低
- 可复用现有代码较多
- 比完整仿真 + SAT 路线更容易做出稳定版本


## 7. 推荐实施路线

比较现实的研发顺序是：

1. 第一阶段实现 `windowed ttopt`
2. 第二阶段在此基础上加入仿真信息做候选筛选
3. 第三阶段再引入 SAT 做精确验证和更激进替换

这种路线的好处是：

- 第一阶段就能较快看到收益
- 后续阶段可以逐步抬高 QoR 上限
- 避免一开始就进入高复杂度实现


## 8. 最终结论

这三种方案里：

- 如果只追求 QoR 上限，`simulation + SAT guided ttopt` 最强
- 如果追求现阶段最值得做、最容易做成有效系统的方案，`windowed ttopt` 最合适
- 如果只想先做一个低风险试验版本，`cut-based partial-support ttopt` 最容易启动

因此，更实际的建议是：

- 以 `windowed ttopt` 作为首个增强方向
- 把 `simulation + SAT` 作为后续增强层
- 把 `cut-based partial-support` 作为低风险基线或辅助模块
